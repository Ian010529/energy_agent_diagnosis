# 代码工程里的数据接入方案

## 一、总体原则

1. 工具接口先稳定，数据源后替换；
2. 业务层只依赖统一 Tool/Provider 接口，不直接依赖具体数据库或外部系统；
3. `Mock` 与 `Real` 使用同一套输入输出 schema；
4. 从第一天开始就保留 provider 抽象层，避免后期大规模重构；
5. 所有真实接入都通过 adapter 完成，不让 Agent 工作流感知底层实现差异。

### 1.1 数据接入子阶段

本文的阶段一和阶段三，与《研发落地补充说明》的阶段二共同组成 Provider 级数据接入子流程：

1. **D1 / 阶段一 Mock**：使用可控样例验证工具、工作流、引用和降级；
2. **D2 / 阶段二 Schema 冻结**：冻结该 Provider 的输入输出、错误码、Trace 和幂等约束；
3. **D3 / 阶段三 Real Adapter**：在不修改上层调用方的前提下替换真实数据源。

D1-D3 不替代实施计划的阶段 0 到阶段 7，也不要求所有 Provider 同步推进。设备画像可以已进入 D3，而工单写入仍停留在 D1；P0/P1 则继续表示一期内部优先级。

| 实施阶段 | 本文子阶段的主要工作 |
|----------|----------------------|
| 阶段 1 | 建立 base interface、Mock 骨架和配置切换能力 |
| 阶段 2 | 设备、告警、时序、手册、工单逐项完成 D1-D3 |
| 阶段 3 | 检索 Provider 完成 Mock、Schema 冻结和真实检索适配 |
| 阶段 4 | 使用完整 Mock 做主链路回归，并联调已就绪 Real Adapter |
| 阶段 5 | 审核、案例和写入类 Provider 进入 D2/D3 |
| 阶段 6 | 对 D3 Provider 做超时、熔断、降级和回归验证 |
| 阶段 7 | 按试点场站灰度启用 Real，并继续替换剩余 Mock |

## 二、工程分层建议

建议代码工程按以下方式组织：

```text
src/
  app/
    api/
    workflow/
    tools/
  domain/
    diagnosis/
    retrieval/
    memory/
  providers/
    device_profile/
      base.py
      mock.py
      real.py
    alarm/
      base.py
      mock.py
      real.py
    timeseries/
      base.py
      mock.py
      real.py
    manual_search/
      base.py
      mock.py
      real.py
    ticket_search/
      base.py
      mock.py
      real.py
    graph_relation/
      base.py
      mock.py
      real.py
    ticket_write/
      base.py
      mock.py
      real.py
    case_review/
      base.py
      mock.py
      real.py
  infrastructure/
    mysql/
    redis/
    milvus/
    neo4j/
    influxdb/
  fixtures/
    devices.json
    alarms.json
    timeseries/
    manuals/
    tickets.json
```

核心约束：

1. `tools/` 只负责 schema 校验、调用 provider、整理返回结构；
2. `providers/` 负责具体数据接入；
3. `workflow/` 不直接访问数据库或第三方 API；
4. `fixtures/` 中的数据结构尽量贴近真实数据结构。

## 三、阶段一：Mock 阶段

### 3.1 阶段目标

Mock 阶段的目标不是模拟所有企业复杂度，而是尽快跑通以下主链路：

1. 输入告警或问题；
2. 查询设备与告警上下文；
3. 拉取时序摘要；
4. 检索手册和相似工单；
5. 生成证据包；
6. 输出候选根因、引用和排查建议；
7. 支持人工补充问题和案例回写。

Mock 是长期保留的回归基线，不因 Real Adapter 上线而删除。每个 Provider 的 Mock 验收通过后进入 D2 Schema 冻结；完成 D2 后才能开发或切换对应 Real Adapter。

### 3.2 为什么必须先做 Mock

1. 真实企业系统依赖多、对接慢，容易阻塞项目启动；
2. 详细设计里最复杂的是工作流和 RAG，不是最先接数据库；
3. 如果没有可运行的 mock 主链路，后续无法验证 Prompt、状态机、引用、降级策略；
4. mock 方案能让同学先做 demo、联调前端、准备面试展示。

### 3.3 Mock 数据源建议

| 工具 | Mock 数据来源 | 推荐格式 |
|------|--------------|---------|
| `get_device_profile` | 本地设备台账样例 | `devices.json` |
| `get_alarm_detail` | 本地告警事件样例 | `alarms.json` |
| `query_timeseries_window` | 预生成时序片段 | `CSV` / `JSON` |
| `search_manual_chunks` | 本地解析后的手册 chunk | `JSONL` / `SQLite` |
| `search_similar_tickets` | 脱敏历史工单样例 | `tickets.json` |
| `query_graph_relations` | 简化关系表 | `graph_relations.json` |
| `create_or_update_ticket` | 本地模拟工单表 | `mock_tickets.db` / `JSON` |
| `append_case_review` | 本地审核记录表 | `mock_reviews.db` / `JSON` |

### 3.4 Mock 数据最小规模建议

为了让主链路“看起来像真的”，建议至少准备：

1. 设备画像：10 到 20 台设备；
2. 告警事件：20 到 50 条；
3. 时序窗口样例：每类核心告警至少 5 组；
4. 手册 chunk：100 到 300 条；
5. 工单案例：30 到 100 条；
6. 图谱关系：20 到 50 条核心关系。

### 3.5 Mock Provider 实现方式

每类 provider 都实现统一接口，例如：

```python
from typing import Protocol


class DeviceProfileProvider(Protocol):
    async def get_device_profile(self, payload: dict) -> dict:
        ...
```

对应 mock 实现：

```python
class MockDeviceProfileProvider:
    def __init__(self, data_path: str):
        self.data_path = data_path

    async def get_device_profile(self, payload: dict) -> dict:
        # 从本地 JSON 中按 device_id 查询
        ...
```

### 3.6 Mock 阶段的工具实现建议

#### 3.6.1 `get_device_profile`

Mock 数据至少包含：

1. `device_id`
2. `device_type`
3. `device_model`
4. `manufacturer`
5. `site_id`
6. `commission_time`
7. `status`

#### 3.6.2 `get_alarm_detail`

Mock 告警至少包含：

1. `alarm_id`
2. `device_id`
3. `alarm_name`
4. `alarm_level`
5. `trigger_time`
6. `raw_message`

#### 3.6.3 `query_timeseries_window`

建议不用一开始就接真 InfluxDB，可以用以下两种方式：

1. 本地 CSV 读入后做窗口聚合；
2. 本地 JSON 直接保存预计算后的时序摘要。

推荐先返回摘要型结果：

1. `metric_name`
2. `min`
3. `max`
4. `avg`
5. `trend`
6. `anomaly_points`

#### 3.6.4 `search_manual_chunks`

Mock 阶段建议直接维护一份解析后的手册 chunk 文件，而不是现场临时解析 PDF。

每条 chunk 最少包含：

1. `doc_id`
2. `chunk_id`
3. `content`
4. `device_type`
5. `device_model`
6. `chapter_title`
7. `page_no`

检索实现可分两步：

1. 先用简单关键词过滤；
2. 再用 embedding 或直接固定排序做演示。

#### 3.6.5 `search_similar_tickets`

Mock 工单应尽量脱敏但保留结构：

1. `ticket_id`
2. `device_model`
3. `alarm_name`
4. `fault_symptom`
5. `root_cause`
6. `action_taken`
7. `is_verified`
8. `close_time`

#### 3.6.6 `query_graph_relations`

Mock 阶段不需要真的部署 Neo4j，也可以先用简化关系表：

```json
[
  {
    "alarm_name": "PCS机柜温度持续升高",
    "component": "散热风扇",
    "fault_cause": "风扇失效",
    "action": "检查风扇转动状态"
  }
]
```

图谱接口、Mock 关系和无结果降级属于基础闭环；Neo4j 真实接入、关系抽取和质量治理属于增强实现。两种形态共用同一 Provider 契约。

#### 3.6.7 `create_or_update_ticket`

Mock 阶段只要做到：

1. 能生成一个模拟工单号；
2. 能把输入写进本地 SQLite 或 JSON；
3. 能在前端上看到“已创建/已更新”状态。

Mock 状态必须明确标记 `source=mock`。基础闭环可以生成草稿或模拟编号；真实创建/更新仍需在 D3 中通过权限、幂等、审计和显式确认门禁。

#### 3.6.8 `append_case_review`

Mock 阶段只需支持：

1. 记录审核人；
2. 记录审核结果；
3. 记录根因与评论；
4. 更新案例状态。

### 3.7 Mock 阶段验收标准

1. 所有工具都能返回与真实 schema 一致的结果；
2. Agent 主链路可在本地完整跑通；
3. Web 端可展示引用、候选根因和补充提问；
4. 至少准备 3 类典型告警演示样例；
5. 能演示“诊断 -> 人工确认 -> 案例沉淀”的闭环。

## 四、阶段三：Real Adapter 阶段

### 4.1 阶段目标

在 Mock 主链路稳定后，将底层数据源逐步替换为企业真实系统，同时保持上层 workflow、tool schema 和前端交互不变。

这一阶段的核心不是重写业务逻辑，而是把：

`MockProvider -> RealProvider`

平滑替换掉。

这里的 Real Adapter 是 D3 成熟度，不是新的全局实施阶段。只要单个 Provider 已完成 D2，就可以在实施阶段 2 到阶段 7 之间按依赖成熟度进入 D3。

### 4.2 Real Adapter 的基本模式

每类工具都保留 base 接口，然后新增真实适配器实现：

```python
class RealDeviceProfileProvider:
    async def get_device_profile(self, payload: dict) -> dict:
        # 调企业 CMDB / 设备台账接口
        ...
```

再通过配置切换：

```yaml
providers:
  device_profile: real
  alarm: real
  timeseries: real
  manual_search: real
  ticket_search: mock
```

这样就能支持分工具渐进替换，而不是一次性全切。

### 4.3 各类 Real Adapter 的典型接入方式

#### 4.3.1 `get_device_profile`

真实来源通常是：

1. CMDB
2. 设备台账系统
3. 资产管理系统

Adapter 要做的事情：

1. 调内部 API 或查 MySQL；
2. 映射企业字段到统一 schema；
3. 补齐标准字段；
4. 屏蔽上游字段差异。

#### 4.3.2 `get_alarm_detail`

真实来源通常是：

1. 告警中台
2. EMS/SCADA 告警系统
3. 监控告警平台

Adapter 重点：

1. 标准化告警名称；
2. 提取原始 payload 的关键信息；
3. 统一时间格式与等级枚举。

#### 4.3.3 `query_timeseries_window`

真实来源通常是：

1. InfluxDB
2. 企业时序平台
3. SCADA 历史库

Adapter 重点：

1. 屏蔽底层 query 差异；
2. 做窗口聚合与异常点提取；
3. 控制超时与最大点数；
4. 返回统一摘要结果，而不是直接把海量原始点给 Agent。

#### 4.3.4 `search_manual_chunks`

真实来源通常是：

1. MinIO 中的原始文档
2. Milvus 向量库
3. OpenSearch/Elasticsearch 关键词索引

Adapter 重点：

1. 按 `filters` 做元数据过滤；
2. 执行 hybrid retrieval；
3. 返回统一的 chunk 结果；
4. 补齐页码、版本、命中分数和来源信息。

#### 4.3.5 `search_similar_tickets`

真实来源通常是：

1. 工单系统数据库
2. 工单检索索引
3. 审核案例主库

Adapter 重点：

1. 从工单系统提取结构化工单；
2. 与 Milvus / 关键词索引结合做检索；
3. 区分已审核和未审核案例；
4. 限制低质量、未闭环工单进入强结果集。

#### 4.3.6 `query_graph_relations`

真实来源通常是：

1. Neo4j
2. 关系图服务
3. 预计算关系表

Adapter 重点：

1. 根据 `alarm_name / component / device_type` 查询关系；
2. 把图查询结果转成统一 JSON；
3. 明确关系置信度或来源案例数量。

#### 4.3.7 `create_or_update_ticket`

真实来源通常是：

1. 企业工单系统 API
2. 内部流程平台

Adapter 重点：

1. 权限校验；
2. 显式确认令牌校验；
3. 创建/更新工单；
4. 返回真实工单号与系统状态。

#### 4.3.8 `append_case_review`

真实来源通常是：

1. 案例主表
2. 审核表
3. 后台运营系统

Adapter 重点：

1. 写审核记录；
2. 驱动案例状态流转；
3. 触发 Milvus / Neo4j 异步更新事件。

### 4.4 Real Adapter 阶段推荐切换顺序

不建议一次性全切真实系统，推荐按业务影响和实现难度逐步替换：

1. `get_device_profile`
2. `get_alarm_detail`
3. `query_timeseries_window`
4. `search_manual_chunks`
5. `search_similar_tickets`
6. `query_graph_relations`
7. `append_case_review`
8. `create_or_update_ticket`

原因：

1. 先切只读工具，风险最低；
2. 再切检索和图谱；
3. 最后切写入类工具，避免误写真实业务系统。

### 4.5 Real Adapter 工程要求

每个真实适配器都必须补齐以下能力：

1. 超时控制；
2. 重试策略；
3. 熔断与降级；
4. trace_id 透传；
5. 审计日志；
6. schema 映射与字段兜底；
7. `NOT_FOUND / PARTIAL_SUCCESS / FAILED` 标准状态返回。

为兼容四份文档中的两类返回示例，Adapter 边界还应接受 `OK/SUCCESS`、`TIMEOUT`、`DEGRADED`，并将顶层 `trace_id/source` 或 `meta.trace_id/source_system/provider_type` 归一化为统一 ToolResult。`alarm_time/trigger_time`、`start/end` 与 `start_time/end_time` 也必须通过映射层兼容，业务层不得感知字段差异。

### 4.6 配置切换建议

建议通过环境变量或配置文件切换 provider：

```yaml
providers:
  device_profile: mock
  alarm: mock
  timeseries: mock
  manual_search: mock
  ticket_search: mock
  graph_relation: mock
  ticket_write: mock
  case_review: mock
```

上线前逐步切换为：

```yaml
providers:
  device_profile: real
  alarm: real
  timeseries: real
  manual_search: real
  ticket_search: real
  graph_relation: real
  ticket_write: real
  case_review: real
```

### 4.7 Real Adapter 阶段验收标准

1. 上层 workflow 无需改动即可切换真实 provider；
2. 所有真实 provider 的返回结构与 mock 完全兼容；
3. 至少 80% 的只读工具完成真实接入；
4. 高风险写工具在权限、审计、确认机制上可控；
5. 出现真实系统故障时，仍能通过降级路径维持基本诊断能力。

“至少 80% 的只读工具”是 D3 阶段的进入试点里程碑，不表示剩余接口被删除：未完成的只读 Provider 继续使用 Mock 或降级结果，并在试点清单中显式标注；被选入真实试点主链路的关键 Provider 必须全部可用。

## 五、推荐落地顺序

从 0 开始做代码工程，推荐实际顺序如下：

1. 先写 base interface；
2. 再写 mock provider；
3. 再把 tool schema 和 provider 接起来；
4. 再接 LangGraph workflow；
5. 再做 Web 端联调；
6. 最后把 provider 一类一类换成真实 adapter。

上述顺序按 Provider 滚动执行，而不是全项目只执行一次。Mock fixtures、Schema 契约测试和 Real Adapter 集成测试三套资产同时保留，确保真实系统故障或字段变化时仍能回归和降级。

最重要的不是“先把所有企业系统接好”，而是“先把工程边界和接口边界搭对”。只要 provider 抽象层设计得对，mock 不是临时凑合，而是后面真实接入的加速器。
