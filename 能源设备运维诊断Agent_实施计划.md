# 能源设备运维诊断 Agent — 实施计划

## 一、实施目标

本实施计划面向“能源设备运维诊断 Agent”一期项目落地，目标是在可控范围内分阶段完成从基础设施搭建、数据接入、RAG 能力建设、Agent 工作流开发到试点上线的完整交付。

一期交付目标聚焦以下能力：

1. 支持 Web 端对话诊断和告警事件触发；
2. 支持设备信息、时序数据、设备手册、历史工单的统一接入；
3. 支持基于 LangGraph 的诊断工作流；
4. 支持带引用的诊断结果输出；
5. 支持低置信度转人工和案例审核沉淀；
6. 支持可观测、可评估、可回滚的试点上线。

## 二、实施原则

1. 先打通主链路，再增强复杂能力；
2. 先做可验证的小范围试点，再逐步扩展设备类型；
3. 先保证数据和证据可靠，再提升模型表现；
4. 先实现人工审核闭环，再考虑自动化沉淀；
5. 每个阶段都必须有可验收输出，避免长期停留在“开发中”状态。

### 2.1 多维推进坐标

四份文档中的“阶段”和“优先级”属于不同维度，全部保留并按下列方式协同：

1. 本文的阶段 0 到阶段 7 是项目整体实施阶段；
2. 数据接入方案中的 D1（Mock）、D2（Schema 冻结）、D3（Real Adapter）是每个 Provider 独立推进的数据接入子阶段；
3. 研发补充说明中的 P0/P1 是任务优先级，不是实施阶段；
4. 各 Provider 可以滚动推进：某个 Provider 完成 D1 和 D2 后即可进入 D3，不要求所有 Provider 同时切换。

| 实施阶段 | 数据接入子阶段映射 | 主要优先级 | 说明 |
|----------|--------------------|------------|------|
| 阶段 0 | D1/D2/D3 前置确认 | P0 | 冻结试点范围、外部系统清单和验收口径 |
| 阶段 1 | D1 公共基础 | P0 | 建立 Provider 抽象、Mock 骨架和统一契约基础 |
| 阶段 2 | 结构化及知识数据的 D1 -> D2 -> D3 | P0 | 先 Mock 验证，再按接口成熟度冻结 Schema 并接入真实数据 |
| 阶段 3 | 检索类 Provider 的 D1 -> D2 -> D3 | P0 | 完成 Mock 检索、契约冻结和真实检索适配 |
| 阶段 4 | 主链路 Mock 完整验收及已就绪 Real Adapter 联调 | P0 | 保留完整 Mock 回归能力，同时接入已成熟真实 Provider |
| 阶段 5 | 审核、案例及写入类 Provider 的 D1 -> D2 -> D3 | P0/P1 | 只读优先，写入能力经过确认后启用 |
| 阶段 6 | D3 稳定性、降级与回归验证 | P0 | 对全部已接入 Provider 做质量门禁 |
| 阶段 7 | Real Adapter 灰度启用与剩余 Provider 迭代 | P0/P1 | 在试点范围内逐步扩大真实数据覆盖 |

## 三、范围边界

### 3.1 一期实施范围

1. Web 端诊断入口；
2. 告警触发后的自动诊断主链路；
3. 设备手册、历史工单、时序数据接入；
4. 混合检索、查询重写、证据重排；
5. 候选根因生成、引用输出、人工补充提问；
6. 会话记忆、案例草稿生成、审核后入库；
7. 基础监控、日志、离线评估和试点验收。

### 3.2 一期不纳入范围

1. 无人值守自动处置；
2. 移动端 APP；
3. 多模态图片识别；
4. 大规模跨设备类型泛化；
5. 不经人工审核的自动知识回写。

### 3.3 范围分层

为同时保留完整架构能力和小范围试点原则，一期范围分为三层：

1. **设计覆盖层**：保留储能 PCS、光伏逆变器、风电关键部件三类业务场景，以及温度、风扇、通讯、功率、传感器等告警类型；
2. **Mock 与评估层**：为上述场景准备可复现样例，用于验证通用 Schema、Provider、RAG 和工作流抽象；
3. **试点验收层**：由阶段 0 从设计覆盖层中书面确认 1 到 2 类设备、1 个场站和 3 到 5 类告警，作为首次上线门槛。

未进入首次试点的设备和告警仍保留在一期设计与 Mock 资产中，不视为被移出一期；其真实接入和上线验收可在阶段 7 滚动完成。

## 四、阶段实施计划

### 阶段 0：项目启动与需求冻结

目标：

1. 明确一期设备范围、试点场站、优先告警类型；
2. 明确外部系统依赖、接口责任人、试点验收口径；
3. 冻结一期范围，避免边做边扩。

核心任务：

1. 确认优先覆盖的设备类型，建议先选 1 到 2 类：
   - 储能 PCS
   - 光伏逆变器
2. 确认优先告警模板，建议先选 3 到 5 类：
   - 温度异常
   - 风扇异常
   - 通讯异常
   - 功率异常
   - 传感器异常
3. 明确外部依赖系统：
   - 时序数据源
   - 设备元数据源
   - 工单系统
   - 告警源系统
4. 确定试点指标：
   - 首轮建议时长
   - 工具调用成功率
   - Top-3 根因命中率
   - 人工介入率
5. 明确实施角色与分工。

产出物：

1. 试点范围清单；
2. 设备与告警优先级列表；
3. 外部接口对接清单；
4. 一期验收指标清单。

验收标准：

1. 范围明确且有书面确认；
2. 外部依赖责任人明确；
3. 一期不做内容有明确边界。

### 阶段 1：基础设施与工程骨架

目标：

1. 搭建项目基础运行环境；
2. 建立服务骨架、配置体系和日志链路；
3. 为后续数据接入与工作流开发提供可运行底座。

核心任务：

1. 搭建基础服务：
   - FastAPI 网关
   - Agent 服务
   - 检索服务
   - 工具服务
   - 记忆服务
2. 部署基础中间件：
   - MySQL
   - Redis
   - MinIO
   - Milvus
   - Neo4j
   - RabbitMQ

完整技术架构还包括详细设计中的 InfluxDB，以及 RAG 关键词检索使用的 OpenSearch 或 Elasticsearch。上述组件与本列表共同构成技术并集：开发环境可按当前阶段启用最小集合，试点环境按实际 Provider 和检索能力启用对应组件。

基础服务名称描述逻辑职责，不要求一期全部独立部署。支持以下物理部署配置：

1. 本地开发配置：检索、工具、记忆逻辑模块可与 Agent 合并；
2. Mock 联调配置：网关、Agent、能力模块可按联调边界拆分；
3. 试点配置：`gateway-service`、`agent-service` 优先独立，retrieval/tool、memory 可按负载合并或拆分；
4. 扩展配置：保留详细设计中的全部逻辑服务，并可增加 admin/evaluation 独立服务。
3. 建立配置体系：
   - 环境配置
   - 模型配置
   - 检索参数配置
   - 权限和角色配置
4. 建立通用能力：
   - Trace ID
   - 结构化日志
   - 健康检查
   - 基础鉴权
   - 错误码规范
5. 建立 CI/CD 基线：
   - 测试
   - 镜像构建
   - 部署脚本

产出物：

1. 可运行的服务骨架；
2. 基础中间件部署环境；
3. 配置中心或配置文件体系；
4. 通用日志与监控埋点框架。

验收标准：

1. 所有基础服务可启动；
2. 核心依赖可联通；
3. 日志、健康检查、配置读取正常。

### 阶段 2：数据接入与知识库构建

目标：

1. 打通一期所需的多源数据接入；
2. 建立设备手册与工单知识库；
3. 形成可供检索链路使用的标准化数据资产。

核心任务：

1. 完成设备元数据接入：
   - 设备类型
   - 设备型号
   - 厂家
   - 场站信息
2. 完成告警数据接入：
   - 告警 ID
   - 告警名称
   - 告警等级
   - 触发时间
3. 完成时序数据接入：
   - 设备温度
   - 电流/电压
   - 风扇转速或状态
   - 环境温度
4. 完成工单数据导入与清洗：
   - 去除低质量模板语句
   - 统一设备与告警术语
   - 标记已审核/未审核状态
5. 完成手册处理流程：
   - 文档上传
   - OCR/解析
   - chunk 切分
   - metadata 生成
   - MinIO 存储
   - Milvus 入库
6. 完成基础关键词索引构建：
   - 手册标题
   - 告警词
   - 工单摘要

产出物：

1. 标准化设备元数据表；
2. 可查询的时序接口；
3. 已清洗的历史工单库；
4. 已入库的设备手册向量与关键词索引。

验收标准：

1. 能根据设备 ID 查询到设备画像；
2. 能根据时间窗拉取关键时序指标；
3. 手册与工单具备可检索性；
4. 数据质量满足最小可用标准。

### 阶段 3：RAG 检索链路实现

目标：

1. 实现可用的混合检索能力；
2. 打通查询重写、过滤、召回、重排和引用生成；
3. 形成稳定的证据包输出。

核心任务：

1. 实现查询重写模块：
   - 规则标准化
   - 设备/告警字典映射
   - Qwen 轻量改写
   - 降级回退
2. 实现混合召回：
   - 手册关键词召回
   - 手册向量召回
   - 工单关键词召回
   - 工单向量召回
   - 图谱补充召回
3. 实现统一重排：
   - score 归一化
   - reranker 接入
   - final_score 计算
   - 去重和多样性控制
4. 实现证据归并：
   - 来源分组
   - 摘要裁剪
   - 引用 ID 生成
5. 建立 RAG 离线评估集并做首轮评估。

产出物：

1. 可独立调用的检索服务；
2. 查询重写与重排模块；
3. 标准证据包结构；
4. 第一版 RAG 评估报告。

验收标准：

1. 给定典型告警问题，能稳定召回相关手册与相似工单；
2. 证据结果可追溯；
3. Top-K 召回质量达到试点可用水平。

### 阶段 4：Agent 工作流与工具编排

目标：

1. 实现从请求进入到结果输出的完整工作流；
2. 打通状态机、工具调用和结果生成；
3. 支持人工补充问题和多轮会话。

核心任务：

1. 落地 LangGraph 工作流：
   - `intent_router`
   - `entity_parser`
   - `plan_builder`
   - `tool_dispatcher`
   - `evidence_aggregator`
   - `gap_detector`
   - `reason_generator`
   - `response_generator`
   - `rule_checker`
2. 实现工具层：
   - 设备画像查询
   - 告警详情查询
   - 时序窗口查询
   - 手册检索
   - 工单检索
   - 图谱查询
3. 实现人工补充提问分支：
   - 缺失信息识别
   - 问题生成
   - 用户回答回写
   - 基于新信息继续推理
4. 实现 Web 端主流程联调：
   - 发起诊断
   - 查看中间状态
   - 查看最终结果
   - 继续追问“为什么”

产出物：

1. 可运行的 Agent 主链路；
2. 标准化工具注册表；
3. Web 端联调版本；
4. 典型场景 Demo。

验收标准：

1. 至少 3 类优先告警可跑通主链路；
2. 工具调用成功率达到目标；
3. 多轮对话与补充提问流程可用。

### 阶段 5：记忆、案例沉淀与人工审核

目标：

1. 实现会话级短期记忆；
2. 实现审核控制下的长期案例沉淀；
3. 打通“诊断结果 -> 案例草稿 -> 审核 -> 入库 -> 再召回”闭环。

核心任务：

1. 实现 Redis 短期记忆：
   - 会话对象
   - 工具缓存
   - 待补充问题
   - 流式恢复
2. 实现案例草稿生成：
   - 从诊断结果抽取结构化字段
   - 生成 `DRAFT`
   - 校验最小字段完整性
3. 实现审核流：
   - `PENDING_REVIEW`
   - `APPROVED`
   - `REJECTED`
   - `DISABLED`
4. 实现索引回写：
   - MySQL 主记录
   - Milvus 索引
   - Neo4j 关系更新
5. 实现失败补偿和回滚：
   - 索引重试
   - 版本替换
   - 案例下线

产出物：

1. 记忆服务；
2. 案例审核后台或管理接口；
3. 审核通过案例的召回闭环；
4. 审计日志。

验收标准：

1. 会话上下文可恢复；
2. 未审核案例不会进入强证据集；
3. 审核通过案例可在后续检索中被召回。

### 阶段 6：评估、风控与稳定性建设

目标：

1. 建立上线前质量门槛；
2. 补齐降级、熔断、权限和审计能力；
3. 形成可监控、可追踪、可回归的交付版本。

核心任务：

1. 建立离线评估：
   - Top-1/Top-3 根因命中率
   - Evidence Coverage
   - Tool Success Rate
   - Human Escalation Precision
2. 建立线上监控：
   - 节点耗时
   - 工具调用成功率
   - 模型时延
   - 会话失败率
3. 实现 Guardrail：
   - 无引用结论拦截
   - 高风险建议人工确认
   - 工具参数 schema 校验
   - 结果完整性校验
4. 完成权限与审计：
   - viewer/operator/reviewer/admin
   - 高风险操作留痕
5. 完成异常与降级验证：
   - LLM 不可用
   - 向量检索不可用
   - 时序接口不可用
   - 图谱不可用

产出物：

1. 上线前评估报告；
2. 稳定性与风险控制清单；
3. 监控大盘和告警规则；
4. 回归测试脚本。

验收标准：

1. 关键指标达到试点门槛；
2. 降级链路经演练可用；
3. 高风险场景不会直接输出不受控结果。

指标并集与门槛管理如下：

| 指标 | 当前口径 | 阶段 0/6 要求 |
|------|----------|--------------|
| 首轮/完整诊断建议时长 | 目标 1 到 3 分钟 | 阶段 0 明确统计起止点和分位数 |
| Tool Success Rate | 不低于 95% | 阶段 6 必须达标 |
| 会话失败率 | 不高于 5% | 阶段 6 必须达标 |
| Evidence Coverage | 不低于 80% | 阶段 6 必须达标 |
| 无引用强结论 | 0 | 阶段 6 必须达标 |
| 高风险建议人工确认覆盖率 | 100% | 阶段 6 必须达标 |
| Top-1/Top-3 根因命中率 | 保留并持续统计 | 阶段 0 由业务方确认试点阈值 |
| Human Escalation Precision | 保留并持续统计 | 阶段 0 由业务方确认试点阈值 |
| 人工介入率 | 运营观察指标 | 阶段 0 明确目标区间，不简单追求越低越好 |

### 阶段 7：试点上线与闭环优化

目标：

1. 在试点场景上线；
2. 收集真实反馈；
3. 基于线上数据做第一轮迭代优化。

核心任务：

1. 选定试点范围：
   - 1 个场站
   - 1 到 2 类设备
   - 3 到 5 类优先告警
2. 进行灰度上线：
   - 内部用户白名单
   - 指定班组使用
   - 只读模式优先
3. 收集反馈：
   - 诊断耗时
   - 召回是否相关
   - 建议是否可执行
   - 人工追问是否过多
4. 进行第一轮优化：
   - 检索参数
   - 模板与 Prompt
   - 工具超时与重试
   - 案例审核规则

产出物：

1. 试点上线版本；
2. 用户反馈清单；
3. 第一轮优化方案；
4. 二期建议列表。

验收标准：

1. 试点用户可稳定使用；
2. 至少完成一个真实告警闭环；
3. 系统输出对运维排查有明确帮助；
4. 具备继续扩展到更多设备类型的基础。

### 4.1 横向能力成熟度

以下能力在不同文档中的定位并不互斥，统一按“基础能力 + 增强实现”理解：

| 能力 | P0/基础交付 | P1/增强交付 |
|------|-------------|-------------|
| 知识图谱 | 保留 `query_graph_relations` 接口、Mock 关系数据和降级路径 | 接入 Neo4j、增强关系抽取和关系质量治理 |
| 工单能力 | 输出创建/升级建议并生成待确认草稿 | 经过权限、幂等、审计和人工确认后写入真实工单系统 |
| 知识库 | 完成文档解析、chunk、索引和查询流水线 | 提供上传、版本、字典和模板管理后台 |
| 可观测性 | Trace ID、结构化日志、节点与工具指标 | 监控看板、策略对比和运营分析 |
| 评估能力 | 基础回归集和上线前质量门禁 | 独立 evaluation-service、A/B 对比和持续评估 |

P1 表示增强优先级，不等同于二期；阶段 7 可根据试点需要交付部分 P1 能力。

2. `EPIC-02`：数据接入与知识库
3. `EPIC-03`：RAG 检索与重排
4. `EPIC-04`：Agent 工作流与工具调用
5. `EPIC-05`：Web 端交互与会话体验
6. `EPIC-06`：记忆、案例沉淀与审核
7. `EPIC-07`：评估、风控与可观测
8. `EPIC-08`：试点上线与运营反馈

每个 EPIC 下建议拆成 3 类任务：

1. 开发任务；
2. 联调任务；
3. 验证任务。

## 六、关键依赖关系

依赖顺序如下：

1. 基础设施完成后，才能稳定推进数据接入；
2. 数据接入和知识库构建完成后，才能验证 RAG；
3. RAG 与工具接口可用后，才能完成 Agent 主链路；
4. 主链路跑通后，才能做记忆、案例沉淀和审核闭环；
5. 评估与风控必须在试点上线前完成；
6. 试点上线必须建立在可观测和可回滚基础上。

关键阻塞项：

1. 时序数据接口可用性；
2. 历史工单质量；
3. 手册解析质量；
4. 设备/告警标准化字典是否完善；
5. 试点业务方是否提供审核资源。

## 七、风险清单与应对

| 风险 | 表现 | 应对措施 |
|------|------|---------|
| 历史工单噪声高 | 检索结果质量差 | 只将已审核工单纳入强证据集 |
| 手册解析质量差 | chunk 语义断裂 | 增加 OCR 质检与人工抽样验收 |
| 时序数据不完整 | 无法支持根因判断 | 明确走降级链路并转人工 |
| 模型输出发散 | 结论不稳定 | 限制 Prompt、加强引用校验、增加规则兜底 |
| 图谱建设成本高 | 进度拖慢 | 图谱仅作为补充能力，非主链路阻塞项 |
| 审核资源不足 | 案例沉淀速度慢 | 先小规模沉淀高价值案例 |
| 需求扩张过快 | 实施周期失控 | 每阶段做范围评审，冻结非一期需求 |
